# 教学服务管理系统：AI 智能指令网关全量创新方案

## 1. 核心愿景：从“菜单驱动”到“意图驱动”
本方案旨在将 `TeachingServiceSystem` 传统的、基于固定 API 接口的查询模式，进化为以 AI 为核心的**“指令化”交互模式**。
- **现状**: 用户需要层层点击菜单，通过固定表单查询数据。
- **未来**: 用户只需发送自然语言指令（如：“统计本学期软件工程专业挂科率最高的前三门课”），系统自动推理并执行。

---

## 2. 总体架构：多代理协同 (Multi-Agent Architecture)
系统由五大核心代理组成，形成一个完整的闭环：
1. **规划代理 (Planner Agent)**: 负责多意图拆解与任务依赖编排。支持 **多意图并行**（如：同时查询统计与清单）。
2. **路由代理 (Router Agent)**: 负责意图分类与 RAG 工具检索。
3. **执行代理 (Execution Agent)**: 混合模式执行（Function Calling 或 Text-to-SQL）。
4. **验证代理 (Validator Agent)**: 核心安全守卫，通过 AST 解析审计 SQL 语义并注入权限。
5. **合成代理 (Synthesis Agent)**: 将原始 JSON 数据转化为有人情味的自然语言回复。

---

## 3. 自然语言指令的“一生” (Workflow Pipeline)
当系统接收到一条指令，将严格执行以下 **8 个阶段** 的流水线：

### 阶段 0：推理与规划 (Multi-Intent Planner)
- **需求拆解**: 对于复杂需求（如“查第一名是谁，并看他选了什么课”），生成任务图 (DAG)。
- **依赖管理**: 识别出 Task B 依赖于 Task A 返回的 `student_id`。

### 阶段 1：意图解析与工具检索 (Intent & Tool Retrieval)
- **四级过滤机制**: 语义预分类 -> 向量匹配工具 -> 槽位提取 -> 歧义消除。
- **RAG 匹配**: 从向量库中寻找最匹配的 **Java Service 接口描述** 或 **SQL 标准样例**。

### 阶段 2：上下文构建 (Context Construction)
- **Schema RAG 修剪**: 系统只提取与当前指令相关的表结构碎片信息，减少 Token 消耗。
- **权限背景注入**: 自动提取当前登录用户的 ID、角色、所属学院。

### 阶段 3：受控生成 (Controlled Generation)
- **混合决策**: 确定是“填写 Function 调用参数”还是“参考 SQL 样例生成新 SQL”。
- **Chain-of-Thought**: AI 内部进行逻辑推导，确保 SQL 逻辑符合意图。

### 阶段 4：安全沙箱审计 (Security Sandbox & Validator)
- **AST 解析**: 解析语法树，验证是否包含非 `SELECT` 操作，校验表名/列名白名单。
- **强制权限注入**: 在 `WHERE` 子句末尾强制拼接用户维度过滤条件（如 `AND department_id = ...`）。

### 阶段 5：安全执行 (Secure Execution)
- **参数化下发**: 使用 MyBatis-Plus 执行过滤后的 SQL。
- **并发与限流**: 设置执行阈值，防止低效查询阻塞系统。

### 阶段 6：数据合成与润色 (Data Synthesis)
- **多任务结果汇总**: 合并统计数据与明细清单。
- **拟人化转换**: 将表格数据转化为逻辑清晰、带有洞察力的文字回复。

### 阶段 7：日志与反馈 (Observability)
- **全链路追踪**: 记录“意图 -> SQL -> 结果”的全环节，用于模型优化。

---

## 4. 核心突破：知识库与元数据设计 (Knowledge Base Design)
为了平衡灵活性与性能，我们在 `echoflow` 向量库中维护三类核心知识资产：

### 4.1 函数元数据层 (Function Metadata)
- **内容**: 每一个带有 `@AIService` 注解的 Java 接口（半写死 SQL 模式）。
- **作用**: 快速判断是否有高性能、高可靠的预定义接口可供调用。

### 4.2 动态 Schema RAG 层 (Schema Fragments)
- **原子单位**: 以“表”为单位。每张表及其主外键关联关系构成一个文档。
- **设计精髓**: AI 不需要“背诵”全库表结构，只需“按需检索”需要的几张表，极大提升寻路准确率。

### 4.3 黄金 SQL 样例层 (SQL Gold Examples)
- **内容**: 经核准的、包含复杂业务潜规则的 SQL 查询模板。
- **核心思想**: 模仿比创作更准确。AI 通过对比样例，学习如何在当前业务场景下编写正确的 `JOIN` 逻辑。

---

## 5. 执行层的三重保险
1. **白名单校验**: 生成的 SQL 中所有的表名、列名必须在 Schema 文档库中存在。
2. **AST 逻辑审计**: 自动拦截超过 5 个 `JOIN` 或无索引全表扫描的危险 SQL。
3. **数据隔离注入**: 在执行前像“贴标机”一样，强行在生成的 SQL 尾部注入该用户的所属权限标签。

---

## 6. 例外处理：高并发场景一致性
**选课逻辑 (Course Selection)** 及涉及强实时竞争的操作不进入 AI 指令网关。
- **原因**: AI 推理延迟 (3-5s) 与抢课的高并发实时性（微秒级竞争）冲突。

---

## 7. 预期收益
- **零接口开发**: 告别“一个查询组合写一个接口”的冗余，一个网关接口搞定所有长尾查询需求。
- **体验跨越**: 自然语言直达数据，打造真正的“对话式教学管理助手”。
- **安全可见**: 所有由 AI 产生的数据库操作，其在沙箱中的审计全过程均可追踪、可审计。
